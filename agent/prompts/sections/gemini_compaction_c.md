你正在将 Articraft **Gemini** 运行历史压缩为**单一交接摘要（handoff summary）**。

## 任务（Task）

- 仅以提供的 `system_prompt` 与 `messages` 为**唯一事实来源**。
- 产出一份**紧凑、损失最小化**的状态交接，供下一轮编码/调试使用。
- **保留**持久性事实与**立即可执行**的未解决状态。
- **移除**重复内容、寒暄、以及**已解决**的死胡同叙述。

## 优先级顺序（Priority order）

1. 用户意图与成功标准（success criteria）。
2. 会改变“何种代码/几何才算有效”的约束（constraints）。
3. 未解决的 compile / debug / tool 失败项。
4. 已从 tools、files、examples 学到且仍重要的发现。
5. 下一轮应采取的**最佳后续动作**。

## 必须保留的事实（Required preserved facts）

- 物体或任务请求，以及定义成功的要求。
- 重要尺寸、几何约束、空间关系、以及影响验收的关键细节。
- 用户给出的约束、排除项、偏好与设计意图。
- 仍能指导实现的示例、文件或 tool 发现。
- 当前 compile/debug 状态，尤其是**未解决**的失败、警告与阻塞项。
- 在**紧接着的下一轮**仍然重要的具体后续上下文。

## 字段含义（Field meanings）

- `task_requirements`：为使本次运行成功，必须**构建、修复或验证**的内容。
- `constraints`：不得违反的**精确限制**，包括尺寸、排除项、偏好、文件/路径限制、接口约束等。
- `tool_findings`：从 tools、files、examples 或先前输出中学到、且**仍然重要**的具体事实。
- `compile_state`：仅保留**未解决**的失败/警告；或在下一轮仍重要的**当前已知良好状态**。尽可能保留文件名、符号与原因。
- `next_steps`：由历史隐含的最小、高价值后续动作。

## 规则（Rules）

- **仅使用**所提供历史中可支持的事实。**不要**臆造缺失细节。
- 若历史含糊或不完整，应**显式保留歧义**，而非猜测。
- 在重要时保留**精确标识符**、尺寸、文件路径、tool 名称、API 名称、以及**带引号的约束原文**。
- 若某尝试已被否决但仍影响决策，保留**结论**，而非完整叙事。
- 每条字符串优先承载**一个具体事实**。
- 避免重复、矛盾与含糊转述。
- 排除已解决的闲聊、礼貌用语与反复重述。
- 若某节无值得延续的内容，返回**空数组**。

## 输出格式（Output format）

- **仅返回 JSON**。
- 使用下列**精确 schema**，**不得**添加额外字段：
  - `task_requirements`：字符串数组（array of strings）
  - `constraints`：字符串数组
  - `tool_findings`：字符串数组
  - `compile_state`：字符串数组
  - `next_steps`：字符串数组
- 每条目应简洁、具体、可独立理解。
- 优先保留**具体事实**，而非散文式描述。
- **不要**包含 markdown 代码围栏（fences）。
