# 类别入选要求

本文定义在**当前仓库分类体系**（`data/categories/`）中**新增或保留**类别时必须满足的门槛。

**类别 ≠ 有趣的 prompt 点子。** 类别是可持续的分类单元，应能支撑：

- 重复的 prompt 批次（`data/batch_specs/`）
- 多条可信、彼此不同但仍属同类的记录
- 明确的 `target_sdk_version` 映射

---

## 核心标准（总览）

一个好的类别应当：

| 维度 | 要求 |
| --- | --- |
| 可表示性 | 能作为可信的关节化对象，用显式 URDF 关节表达 |
| 可构建性 | 能用当前 `sdk` **稳定**搭建 |
| 特异性 | 对象身份清晰，人一眼能认出「是哪一类东西」 |
| 广度 | 不止支持「某一个」一次性设计 |
| 区分度 | 与已有类别明显不同 |

**任一维度不达标 → 很可能不是正确的分类单元**（应拒绝、合并或改名）。

---

## 1. 关节化适配（Articulation Fit）

类别必须能**诚实**映射到本仓库支持的关节化模型：

- 刚性部件 + **显式关节**连接
- 运动类型：`revolute`（有限转动）、`continuous`（无界旋转）、`prismatic`（平移）、`fixed`（刚性固定）
- **树形**运动学（每个部件恰好一个父关节，无环）

### 应拒绝（定义性行为依赖下列机制时）

| 机制类型 | 为何难诚实表达 |
| --- | --- |
| 闭链运动学环（环为类别核心时） | URDF 树不允许环 |
| 缆绳、皮带、链条耦合 | 无 joint-drives-joint |
| 滑轮路由 | 同上 |
| 需联动约束的齿轮啮合 | 各关节独立运动 |
| 以螺旋螺纹为主定义的运动 | 非单轴 prismatic/revolute 语义 |
| 棘轮、分度、间歇锁定为主行为 | 无棘轮/索引原语 |
| 仅靠仿真「技巧」的接触驱动运动 | 非显式关节 |

### 高风险/通常不合格示例

- 真正的四连杆或剪刀机构（闭链即卖点）
- 棘轮棘爪装置
- 日内瓦机构（Geneva drive）
- 缆绳滑轮升降
- 差速器
- 真实凸轮-从动件系统（凸轮几何不产生运动学）

> **原则：** 若必须「伪造」类别定义性运动，则**不应**设立该类别。

---

## 2. SDK 适配（SDK Fit）

类别须能用仓库**已有**工具与几何模式实际创作。

### 通常良好的特征

- 明确的刚性构件：外壳、导轨、轮毂、支架、叉架、臂、托盘、面板、框架
- 硬表面几何，支撑关系可读
- 清晰的 parent/child 结构
- 可由 primitive、型材挤出、扫掠、旋转体、布尔、直接 CadQuery 实体构建

### 通常不良的特征

- 软体行为为主
- 布料、缆线、弹簧、链条或可变形材料作为**身份核心**
- 高度有机的自由曲面为主视觉
- 依赖 exotic loft 车身等易碎曲面讲故事
- 即使用心 prompt 仍在 compile/QC/测试中脆弱

> **原则：** 若在 compile、QC、测试中即使用心 prompt 仍脆弱 → 不适合作为本仓库类别。

---

## 3. 类别范围（Category Scope）

类别应是**稳定的对象类或机构类**，既不是含糊大伞，也不是微观变体伪装。

### 良好范围

- 清晰命名**一类**连贯对象或机构
- 多个示例共享可识别的结构
- 可在不改变核心身份的前提下做多种 prompt 变体
- 具可预期的「常识关节」集合

### 不良范围

| 问题 | 表现 |
| --- | --- |
| 过宽 | 混合多个对象族，prompt 与拓扑差异巨大 |
| 过窄 | 实质上只有一个具体设计实例 |
| 过模糊 | 输出拓扑/支撑结构 wildly 不一致 |

**自检：** 能否想象**多个**仍读作同一类别的可信示例？若不能，范围错误。

---

## 4. 区分度（Distinctness）

新类别必须在分类法中**挣得位置**，不能只是现有类别的改名或薄扩展。

### 通常应拒绝

- 现有类别 + 一个额外关节
- 现有拓扑的「数量变体」
- 现有类别的子集
- 现有机构的附件变体
- 与已有对象类仅标签不同
- 与多个现有类别高度重叠的伞形名称

### 强新增通常至少带来其一

- 显著不同的关节化拓扑
- 显著不同的支撑结构
- 显著不同的机构原型（archetype）
- 清晰不同的对象身份
- 值得单独维护 prompt 策略的持久类别

---

## 5. 在当前仓库中的落位

从**当前类别元数据**思考，而非遗留 tracker 块。

接受一个类别时，应能合理分配：

- 稳定 `slug`（目录名，如 `washing_machine`）
- 可读 `title`（显示标题）
- 连贯的 prompt 批次策略（见 [CATEGORY_PROMPT_GUIDE_c.md](CATEGORY_PROMPT_GUIDE_c.md)）
- `target_sdk_version`

**当前默认映射：**

- 对象优先、产品身份/外壳/成品外观比「纯 CAD 机构细节」更重要的类别 → 使用 `base`
- **本仓库当前所有类别**均使用 `base`

## 决策清单（新增前逐项打勾）

1. 定义性运动是否可用显式 URDF 关节 + 树形运动学表示？
2. 能否用 `sdk` **干净地**构建多个可信示例？
3. 类别范围是否稳定、连贯？
4. 是否与 `data/categories/` 下已有类别**有意义地区分**？
5. 能否根据几何应如何创作，指定清晰的 `target_sdk_version`？
6. 在多轮 prompt 批次与多条 accepted 记录之后，该类别是否**仍然**有意义——而非只靠一条幸运 prompt？

**任一为「否」→** 通常应拒绝、合并进现有类别，或改名为更好的分类单元。

---

## 实用默认原则

- 优先**耐久**的对象类或机构类，而非花哨一次性概念。
- 优先支撑结构清晰、常识关节明确的类别。
- 优先能产出**多条**好 prompt 的类别，而非单条幸运 prompt。
- 优先主几何可用**当前 SDK 栈**实现的类别。
- **存疑时：** 选更简单、边界更清晰的类别划分。
